---
title: Context Window 设计
description: 在 Context Window 限制内设计 Prompt 与 Task Slice 的方法。
---

## 1) 问题定义：Context Window 不是“容量”，而是“约束系统”

Context Window 不只是 token 上限，更是指令优先级、注意力分配和成本结构的综合约束。  
超限不是唯一问题，常见情况是“未超限但质量退化”，例如指令被稀释、关键信息埋在中段（lost-in-the-middle）。

## 2) 设计目标：把 token 当预算来管理

- 保留“不可丢信息”：系统指令、任务目标、强约束条件；
- 控制“高变信息”：用户上下文、检索片段、临时中间结果；
- 压缩“低价值信息”：冗余对话历史、重复日志、无效失败尝试。

## 3) 实践策略

### 指令分层

- 系统层：角色、边界、输出格式、禁止项；
- 任务层：当前目标、输入摘要、验收标准；
- 证据层：检索结果、日志片段、结构化数据。

### 历史管理

- 对话历史采用“滚动摘要 + 关键锚点”而非全文回放；
- 对失败尝试只保留“失败原因 + 不可行路径”，减少重复踩坑；
- 明确历史有效期，过期信息进入归档，不再默认注入。

### 预算配比（可作为初始模板）

- 15%：系统与约束；
- 25%：当前任务与验收标准；
- 45%：证据与检索片段；
- 15%：中间推理与输出缓冲。

## 4) 失败模式与修复

- **症状：答非所问**  
  原因：同一轮混入多个目标。  
  修复：任务切片，一轮只验证一个结果。

- **症状：格式正确但事实错误**  
  原因：证据不足或证据冲突。  
  修复：先做证据筛选与来源排序，再生成答案。

- **症状：偶发退化、难复现**  
  原因：上下文注入顺序不稳定。  
  修复：固定模板顺序，关键字段结构化。

## 5) 落地检查清单

- 是否定义了 token 预算和注入优先级；
- 是否有历史摘要策略与过期清理策略；
- 是否能在日志中复原“当次输入上下文”；
- 是否对超长输入提供降级路径（摘要、分段、拒答）。
